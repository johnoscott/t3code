## Walkthrough, chapter by chapter

### Ch. 1 — The four shifts (read this twice)

The single most valuable page. If the rest of the doc disappeared, these four bullets would still get you most of the way:

1. **Terminal output → stream of typed events.** Never parse TTY bytes. Consume structured SDK messages. In Swift: a `Codable enum SDKMessage` you switch over. The moment you think "parse this line of output" you're lost.
2. **Interactive prompts → request/response with IDs.** A Claude approval isn't a blocking `y/n`; it's `request { id }` → your UI → `response { id, decision }`. Same pattern for plan-mode acceptance, ask-user-question, etc.
3. **Ephemeral session → durable event-sourced thread.** Every meaningful thing is written as an event the instant it happens. The UI reads a projection over those events. Crash recovery = replay.
4. **Local → client/server, even for single-user.** Because SDK lifecycle ≠ UI lifecycle, because multi-device, because background work, because sandboxing. This is the shift SwiftUI devs usually push back on — it's worth it.

### Ch. 2 — Why the server exists

Answers the obvious pushback to shift #4. A diagram of the four server jobs: transport → orchestration → reactors → adapters. The punchline: every layer is independently testable, swappable, debuggable. You'll build them in that same shape in Swift.

### Ch. 3 — Orchestration model in plain language

The chapter that demystifies the event-sourcing vocabulary T3 uses. Plain definitions of command / event / decider / projection / projector / reactor, plus a direct mapping table:

| T3 concept | TCA concept            |
|------------|------------------------|
| Command    | Action                 |
| Event      | Action (persisted)     |
| Decider    | Reducer (pure portion) |
| Projection | State                  |
| Reactor    | Effect                 |

If you've written a TCA app, you already know this model. The only new idea: **events are durable, separated from commands, so you can replay without re-running commands.** That's what makes crash-recovery trivial.

### Ch. 4 — The end-to-end turn trace ⭐

10 steps following one prompt from keypress to checkpoint. If you read nothing else, read this:

1. Keypress in composer
2. Client dispatches `thread.turn.start` command (one RPC, not per-method)
3. Decider emits `thread.turn-start-requested` event; projection updates
4. `ProviderCommandReactor` sees the event, calls `ClaudeAdapter.startSession` if needed, then `sendTurn`
5. SDK iterator yields `SDKMessage` values; adapter converts to runtime events
6. `ProviderRuntimeIngestion` turns runtime events into domain events (`thread.message.delta`, `thread.activity.*`, etc.)
7. Subscribed clients receive pushes
8. SwiftUI re-renders off the projection
9. Approval: `canUseTool` parks a `Deferred`; UI renders a card; user clicks; command dispatched; deferred resolves; SDK continues
10. Turn ends → `CheckpointReactor` captures git snapshot + computes diff

This same 10-step loop covers every interaction in T3. Internalize it.

## Second half of the walkthrough

### Ch. 5 — Terminal feature → GUI feature mapping

This chapter is mostly a reference table. The underlying principle: **every CLI affordance that used to rely on being in a terminal becomes a typed object + a dedicated view.** "Press y/n" becomes an `Approval` object with an id. "Here's your scrolling output" becomes a `Timeline` that knows about streaming text blocks. `Ctrl+C` becomes a Stop button that dispatches `thread.turn.interrupt`.

Two subtleties worth pulling out:

- **The agent's Bash tool vs. the user opening a terminal are different systems.** The first is a tool call modeled as an activity. The second is a PTY session owned by a Terminal service. Don't unify them; they have different lifecycles and permissions.
- **Plan mode is bidirectional.** The SDK emits `ExitPlanMode` with a proposed plan. Your UI shows it, user decides, you send the decision back. Treat the plan as an object, same as an approval — it's not inline text.

### Ch. 6 — SwiftUI translation

This is where the rubber meets the road. The relevant decisions:

**State management.** `@Observable` is fine to start. If you want discipline and already know TCA, go TCA — the orchestration→TCA mapping in Ch. 3 isn't metaphorical, it's a direct port.

**WebSocket transport.** `URLSessionWebSocketTask` wrapped in an `actor`. The resilient loop pattern (`connecting → open → reconnecting → closed`) and the auto-resubscribing subscription are not optional — they're what makes the app not lose data on wifi drops. Build the reconnect-with-backoff + sequence-gap detection early; retrofitting later is painful.

**Shared contracts (the annoying one).** This is the one cross-language friction point. Your options, in order of how I'd pick them:

- *MVP:* hand-maintain Swift mirror types. Faster to start. Drift is the risk.
- *When the schema stabilizes:* codegen. Either OpenAPI/JSON Schema → Swift, or Protobuf if you're willing. T3 can emit JSON Schema from its Effect schemas (`packages/shared/src/schemaJson.ts`).

**Persistence.** GRDB + SQLite for the event log. Schema is trivial: `events(thread_id, sequence, type, payload_json, created_at)` with a compound index on `(thread_id, sequence)`. Project at launch, optionally snapshot every N events. SwiftData works if you prefer the modern stack, but events get touchy to migrate — keep the schema stable.

**Rendering.** `LazyVStack` in a `ScrollView` for the timeline, or `List` with row identities `(turnId, blockId)`. Don't render all of a transcript as one `Text` — split per block so SwiftUI diffs cheaply. `AttributedString(markdown:)` covers 80% of markdown; `swift-markdown` if you hit edge cases.

### Ch. 7 — The SDK bridge problem ⭐

This is the decision that unlocks everything else for you. Swift has no Anthropic SDK, so you have to choose how your Swift code actually reaches Claude.

Four options, ranked:

1. **Spawn `claude` in headless streaming mode.** `Process` with pipes; `claude --print --output-format stream-json --input-format stream-json`. You parse NDJSON `SDKMessage` lines, write new user messages as JSON to stdin, handle approvals via the CLI's approval channel (check `claude --help` for the current flag — it evolves; last I knew it was something like `--permission-prompt-tool`). This is **exactly** what T3's server is doing, one layer deeper: T3 calls the JS SDK, the JS SDK spawns the CLI, the CLI talks to the model. You're just cutting out the middle layer.

    *Why recommended:* works today, no Node dependency, you learn the message shape directly. If a new SDK message type ships, you just add a case.

2. **Ship a Node sidecar.** Your SwiftUI app bundles a tiny Node process that uses the official SDK and speaks a trivial JSON protocol back to Swift. More dependencies; more features; slightly heavier app bundle. A fine escape hatch if Option A's CLI flags prove too limited.

3. **Point at T3's existing server.** Skip building a backend entirely; implement only the SwiftUI client, speaking the Effect-RPC protocol to the existing Node server. Upside: all four providers (Claude/Codex/Cursor/OpenCode) work for free. Downside: you have to reimplement the Effect-RPC wire format in Swift. Doable — it's JSON — but nontrivial, and it couples you to T3's roadmap.

4. **Reimplement the SDK in Swift.** Don't. You will chase upstream forever.

**My recommendation for your first month: Option A.** Prove the pipe, render messages, handle one approval — end to end. You'll know the message shape cold. Then decide between B (more depth on Claude) and C (multi-agent via T3).

### Ch. 8 — The 10-step build order

This is the part to bookmark and work through. Each step is shippable on its own and gives you a concrete demo to show:

1. **The pipe.** Swift `Process` spawning `claude -p --output-format stream-json`, printing every message. No UI. The moment you see an `assistant` message with streaming text blocks, you've unlocked the architecture.
2. **Minimum SwiftUI timeline.** Prompt field + `List` of streaming assistant text. 200 lines of Swift.
3. **Tool call rendering.** Port T3's `classifyToolItemType` table. Per-class icons, expandable output panel.
4. **Approvals.** Approval card in the timeline, decision sent back over the CLI's approval channel. First feature that feels "GUI-native" vs. "terminal port."
5. **Event log + projections.** GRDB. Quit, reopen, see the conversation.
6. **Session resume.** Store Claude's session id; pass `--resume` on next start. You're now ahead of most CLI users.
7. **Checkpoints + diffs.** Hidden git refs before/after each turn. "Revert this turn" button. This feature sells the app.
8. **Split client/server.** Move the `Process`-based adapter into a Swift backend; connect SwiftUI over WebSocket. Now your iPhone can talk to it.
9. **Multi-thread sidebar.** List of threads, creation, per-thread subscriptions.
10. **Polish.** Markdown, syntax highlighting, keybindings, command palette, search.

Steps 1–4 can fit in a weekend each. 5–7 take longer. 8 is the biggest architectural lift (that's when you stop fighting the "server feels like overkill" urge and actually build it). 9–10 are ongoing.

### Ch. 9 — Pitfalls T3 paid for, so you can skip them

The list is in the doc; the three I'd emphasize:

- **Sequence numbers are not optional.** Tempting to skip for MVP. Don't. The first time your phone wakes from sleep and silently drops 40 events, you'll wish you had them. Add sequence on events from day one; add a `replayEvents(fromSequence:)` RPC as soon as you have any subscription.
- **Approvals are records, not modals.** A pending approval has an id and a persistent record. The UI reflects it. If you model it as a SwiftUI `.sheet` binding, you've built something that breaks on reconnect, on multi-device, on "I want to revisit what I approved earlier." Record first, sheet second.
- **Keep the decider/projector pure.** No `Process`, no network, no file I/O inside those functions. All side effects live in reactors. This is what makes replay, tests, and "why did my state end up like this" tractable. The moment you sprinkle I/O into the decider you have a debugger's nightmare.

Two more worth calling out from the doc: **adapter interface from day one** (even if you only ship Claude, the `ProviderAdapter` protocol lets you add Codex or Cursor in days, not weeks), and **every turn ends with a typed "quiesced" signal** (don't make the UI guess when work is done; emit a signal from the last reactor).

## What I'd do tomorrow if I were you

1. Open Xcode, new macOS SwiftUI app.
2. Run `claude --help` and find the current headless flags. Note the exact spelling.
3. Spawn the subprocess from Swift. Log every NDJSON line to the console.
4. Make one `SDKMessage` Codable enum that decodes whatever comes out. Start with `system` / `assistant` / `user` / `result`; add cases as you hit them.
5. When you see a streaming assistant text block render to the console, you're unblocked. Everything else is UI and plumbing you already know how to build.

The first real decision point after that is the shape of your `SDKMessage` enum and how granular you make the runtime-event → domain-event translation (T3's `ProviderRuntimeIngestion.ts` is the reference implementation to study).
