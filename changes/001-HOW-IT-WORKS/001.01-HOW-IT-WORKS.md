# How It Works — Findings

Response to [`001.00-PROMPT.md`](./001.00-PROMPT.md).

This document traces how T3 Code wraps Claude Code (`@anthropic-ai/claude-agent-sdk`) to produce a browser GUI instead of a terminal experience. File:line references are included so every claim is verifiable — if a reference drifts, the description may have drifted with it.

## TL;DR

- **Three layers, not two.** GUI ↔ T3 **server** ↔ Claude Code SDK. The server is the real wrapper; the GUI is a thin client over an Effect-RPC WebSocket.
- **No terminal is wrapped.** The server imports `@anthropic-ai/claude-agent-sdk` as a library and calls `query()` in-process. It never spawns a `claude` CLI to parse terminal output. The SDK may itself invoke a Claude Code binary (`pathToClaudeCodeExecutable`), but from T3's perspective the surface is an async iterator of structured `SDKMessage` objects.
- **Two surfaces cross the wire.** Commands go *in* through `orchestration.dispatchCommand` (one RPC); events come *out* through subscription streams (`subscribeThread`, `subscribeShell`, etc.). The client does not call `providers.startSession` or similar — that surface appears in some older docs but isn't in the code.
- **CLI-only Claude Code features are re-rendered as GUI affordances.** Tool approvals, streaming output, slash commands, session resume, and status spinners all have dedicated server events and UI components; none of them come from parsing terminal output.

│  - OrchestrationEngine          (event-sourced domain core)  │
│  - ProviderService              (adapter dispatch)           │
│  - ClaudeAdapter / CodexAdapter / CursorAdapter / OpenCode…  │
│  - Reactors (runtime ingest, command, checkpoint, deletion)  │

For comparison, the Codex path uses JSON-RPC over stdio against a `codex app-server` subprocess (via `packages/effect-codex-app-server`). Cursor and OpenCode use ACP (via `packages/effect-acp`). Only Claude is purely in-process SDK calls.

## 1. How the GUI frontend talks to the server

### Transport

- Plain WebSocket. **Effect's RPC framework** (`effect/unstable/rpc` + `RpcSerialization.layerJson`) does the framing and encoding — not a hand-rolled JSON-RPC protocol.
- Client: `apps/web/src/rpc/protocol.ts` builds an `RpcClient.make(WsRpcGroup)` over an `@effect/platform/Socket` WebSocket.
- Server: `apps/server/src/ws.ts` registers handlers via `WsRpcGroup.toLayer(...)` at line ~132 (file is 1,090 lines).
- Shared contracts live in `packages/contracts/src/rpc.ts` (plus `orchestration.ts`, `ws.ts`, `provider.ts`, `providerRuntime.ts`, etc.).

### Method catalog (real, not the stale `.docs/provider-architecture.md` names)

Everything is defined with `Rpc.make(WS_METHODS.x, { payload, success, error })`. A partial list of what actually exists in `packages/contracts/src/rpc.ts`:

Commands & queries:
- `orchestration.dispatchCommand` — **the single entry point for every mutation touching a project/thread/session** (create thread, start turn, interrupt, rollback, approval response, runtime-mode change, etc.)
- `orchestration.getTurnDiff`, `orchestration.getFullThreadDiff`, `orchestration.replayEvents`
- `server.getConfig`, `server.updateSettings`, `server.refreshProviders`, `server.upsertKeybinding`
- `projects.searchEntries`, `projects.writeFile`, `filesystem.browse`
- `git.*` (pull, refresh, stacked action, PR prep, worktree create/remove, branch create, checkout, init)
- `terminal.open | write | resize | clear | restart | close`
- `shell.openInEditor`

Subscriptions (server → client streams):
- `orchestration.subscribeShell` — project and thread list changes
- `orchestration.subscribeThread` — per-thread snapshot + live events
- `orchestration.replayEvents(fromSequenceExclusive)` — gap recovery
- `subscribeGitStatus`, `subscribeTerminalEvents`
- `subscribeServerConfig`, `subscribeServerLifecycle`
- `subscribeAuthAccess`

> There is no `providers.startSession`, `providers.sendTurn`, etc. The agent event flow uses orchestration commands (e.g. `thread.turn.start`, `thread.approval.respond`) dispatched through the single `dispatchCommand` RPC. `AGENTS.md`, `.docs/provider-architecture.md`, and parts of `.docs/architecture.md` still reference the old names — treat them as stale.

### Sequences, snapshots, and gap recovery

Every orchestration domain event carries a monotonically increasing `sequence: number`. `subscribeThread` emits a `{ kind: "snapshot", snapshot, snapshotSequence }` envelope followed by live events with `sequence > snapshotSequence` (see `apps/server/src/ws.ts` around line 686).

Client-side recovery lives in `apps/web/src/orchestrationRecovery.ts`:
- `sequence <= latest` → ignore (duplicate after reconnect)
- `sequence === latest + 1` → apply directly
- `sequence > latest + 1` → gap detected → call `orchestration.replayEvents(latest)` to fetch missed events

This is how reconnects don't lose updates and don't require re-loading the whole thread.

### Connection state machine

`apps/web/src/rpc/wsTransport.ts` (287 lines):
- States: `connecting` → `open` → `reconnecting` → `closed` / `disposed`
- Reconnect uses exponential backoff (capped at 7 retries in `wsConnectionState.ts`)
- Subscriptions live in a self-healing loop: on error it sleeps and re-runs; on success it stays open; on session swap it seamlessly reattaches
- An `onResubscribe` callback is available so consumers can reset UI state (e.g. discard stale snapshots)

### Auth

Defined in `packages/contracts/src/auth.ts` and enforced in `apps/server/src/auth/`:

Bootstrap credentials (one-shot):
- `desktop-bootstrap` — trusted handoff from the Electron shell
- `one-time-token` — a pairing token shown as text/QR when running `t3 serve`

Session credentials (long-lived, returned after bootstrap exchange):
- `browser-session-cookie` (HttpOnly)
- `bearer-session-token` (for non-cookie clients)

WebSocket auth: before upgrading, clients POST `/api/auth/ws-token` (already authenticated via cookie/bearer) and then connect as `ws://host/ws?wsToken=…`. Implementation in `apps/server/src/auth/Layers/ServerAuth.ts` (around line 344-371). Server auth mode is one of `desktop-managed-local`, `loopback-browser`, `remote-reachable`, or `unsafe-no-auth`.

## 2. The Claude Code adapter: where terminal turns into API

File: `apps/server/src/provider/Layers/ClaudeAdapter.ts` (3,217 lines). This is the wrapper.

### SDK surface

```ts
// line 8-21
import { query, /* types */ PermissionMode, CanUseTool, ... }
  from "@anthropic-ai/claude-agent-sdk";
```

The adapter calls **`query(options)`** once per session. `query()` returns a `ClaudeQueryRuntime` — an async iterable of `SDKMessage` values plus imperative controls:
- `setModel(apiModelId)` — switch model mid-session (line 3013)
- `setPermissionMode(mode)` — switch between `default` / `plan` / `bypassPermissions` (lines 3029–3035)
- `interrupt()` — cancel the in-flight turn (line 3091)

Options passed to `query()` include (around line 2849):
- `pathToClaudeCodeExecutable: claudeBinaryPath` — resolved from T3's Claude-Code CLI discovery
- `persistSession: false` — T3 owns persistence, the SDK does not
- `canUseTool` — our approval callback (see §4)
- `abortController` — tied to the session fiber's scope
- `settingSources` — Claude settings source routing

The async iterable is consumed via `Stream.fromAsyncIterable(...)` and drained in a forked Effect fiber (`runSdkStream`, around line 2329).

### Not a subprocess of ours

The SDK may internally invoke a Claude Code binary, but from T3's perspective there is no stdio, no terminal, and no text parsing. All interaction is typed `SDKMessage` objects coming back over an iterator. Any feature that cannot be expressed in the SDK message stream (for example arbitrary human-readable CLI output) simply doesn't exist for the GUI.

### What each method does

| Method                | Line      | Effect                                                                                   |
|-----------------------|-----------|------------------------------------------------------------------------------------------|
| `startSession`        | ~2465     | Builds `ClaudeSessionContext`, calls `query()`, forks `runSdkStream` fiber              |
| `sendTurn`            | ~2998     | Enqueues an `SDKUserMessage` on the prompt queue; emits `turn.started`; may call `setModel` |
| `interruptTurn`       | ~3091     | Awaits `query.interrupt()`                                                               |
| `stopSession`         | ~3152     | Marks context stopped, closes prompt queue, cancels pending approvals, interrupts fiber  |
| `readThread`          | ~3101     | Snapshots `context.turns` — read-only                                                    |
| `rollbackThread`      | ~3108     | Truncates turns locally and updates resume cursor                                        |
| `respondToRequest`    | ~3118     | Resolves the `Deferred<ProviderApprovalDecision>` backing a pending approval             |

### Session state + resume

Session state lives in memory (`sessions: Map<ThreadId, ClaudeSessionContext>`, ~line 980). Resume across restarts is handled by a **`resumeCursor`** the adapter updates after each assistant message (`updateResumeCursor`, ~line 1048). Shape: `{ threadId, resume: <session uuid>, resumeSessionAt: <assistant uuid>, turnCount }`. The cursor is persisted server-side with thread state; on restart, `startSession` is called again with `resumeCursor` in the input and the SDK resumes mid-conversation.

Native SDK event traces are written to NDJSON via `EventNdjsonLogger` (`apps/server/src/provider/Layers/EventNdjsonLogger.ts`), with rotation at 10MB × 10 files.

### Streaming model → orchestration events

Inside `handleStreamEvent` (~line 1552), `content_block_delta` messages become:
- `stream kind = "assistant_text"` for text deltas → `content.delta` event (`itemId`, incremental text)
- `stream kind = "reasoning_text"` for thinking deltas → same channel, different kind
- `content_block_stop` → `completeAssistantTextBlock` emits a completion event

These adapter-level events are pushed into a `runtimeEventQueue`, then `ProviderService` fans them to `streamEvents`, then `ProviderRuntimeIngestion` (`apps/server/src/orchestration/Layers/ProviderRuntimeIngestion.ts`) turns them into orchestration commands (e.g. `thread.message.delta`, `thread.message.appended`), which become domain events, which get projected into the read model, which get pushed to the browser as `orchestration.subscribeThread` stream entries.

## 3. Which Claude Code capabilities are exposed through the GUI

Short answer: **tools, approvals, sessions, resume, streaming, slash commands, hooks, runtime/permission modes — yes; MCP servers and Claude settings — mostly passthrough.** Detail:

### Tool calls

Classification in `classifyToolItemType` (ClaudeAdapter.ts ~line 395):

| Tool pattern                              | `itemType`              |
|-------------------------------------------|-------------------------|
| `bash` / `command` / `shell`              | `command_execution`     |
| `edit` / `write` / `file` / `patch` / …   | `file_change`           |
| `mcp*`                                    | `mcp_tool_call`         |
| `task` / `agent` / `subagent`             | `collab_agent_tool_call`|
| `image`                                   | `image_view`            |
| anything else                             | `dynamic_tool_call`     |

These flow into `item.started` / `item.updated` / `item.completed` runtime events. In the UI they become rows in the work log (`apps/web/src/session-logic.ts` `deriveWorkLogEntries` ~line 472 and `apps/web/src/components/chat/MessagesTimeline*`).

### Approvals (tool permissions)

Claude's `canUseTool` callback is implemented at ClaudeAdapter.ts ~line 2652 (`canUseToolEffect`):

1. If the thread's runtime mode is `full-access`, return `{ behavior: "allow" }` immediately.
2. Otherwise, emit a `request.opened` event with a fresh `requestId` and the tool name/input.
3. `Deferred.await` on a per-request `Deferred<ProviderApprovalDecision>`.
4. When the browser user decides, the GUI dispatches a `thread.approval.respond` orchestration command (`apps/web/src/components/ChatView.tsx`), which the server reactor forwards to `ClaudeAdapter.respondToRequest` (~line 3118), resolving the deferred.
5. The deferred's value becomes the `PermissionResult` returned from `canUseTool` to the SDK — either `{ behavior: "allow", updatedInput? }` or `{ behavior: "deny", message? }`.

Approval UI: `apps/web/src/components/chat/ComposerPendingApprovalPanel.tsx` + `ComposerPendingApprovalActions.tsx`. Buttons are **Cancel turn / Decline / Always allow this session / Approve once**, mapped to the approval decision.

### Special tools with bespoke UI routing

- **`AskUserQuestion`**: routed as a `user-input.requested` event and completed via `respondToUserInput` (ClaudeAdapter.ts ~line 2528). Lets the model ask the user a question without conflating with approvals.
- **`ExitPlanMode`**: routed as `turn.plan.proposed` and denied back to the SDK with a message (~line 2672) — the plan itself is surfaced to the UI so a human can accept/reject it explicitly.

### Hooks

Hook lifecycle messages from the SDK (`hook_started`, `hook_progress`, `hook_response`) are translated into `hook.started` / `hook.progress` / `hook.completed` events (ClaudeAdapter.ts ~line 2072). These render as work-log entries in the timeline. Configuration of hooks themselves is whatever the user has in their Claude settings — T3 doesn't inject or override hooks.

### Slash commands

Two things share the name:

1. **Composer slash commands** (`apps/web/src/composer-logic.ts`): `/model`, `/plan`, `/default` — UI affordances, interpreted entirely in the browser (model picker, plan mode, default mode).
2. **Provider slash commands**: the adapter probes Claude's own slash-command catalog via `probeClaudeCapabilities` (ClaudeAdapter.ts ~line 506) by reading `initializationResult().commands`, then returns them in the provider snapshot (`ClaudeProvider.ts` ~line 60). The composer command menu (`ComposerCommandMenu.tsx`) groups them under "Provider slash commands". Execution still happens inside the SDK/CLI — T3 doesn't re-implement command semantics.

### MCP servers

MCP servers are discovered by the SDK/CLI, not configured by the adapter. MCP tool invocations show up as `mcp_tool_use` content blocks and are rendered with `itemType: "mcp_tool_call"`. The GUI does not currently manage MCP server installs / configuration.

### Runtime and permission modes

Thread-level, not global. The `Thread.runtimeMode` value (`full-access` or `approval-required`) is stored in orchestration state and drives the `canUseTool` short-circuit above. The default is `full-access` (`DEFAULT_RUNTIME_MODE` in `apps/server/src/orchestration/Layers/ProviderCommandReactor.ts` ~line 77). Changing mode is a `thread.runtime-mode-set` orchestration command.

Mapping T3 modes → SDK `PermissionMode` (ClaudeAdapter.ts ~line 2836):

| T3 runtime mode         | Claude SDK `PermissionMode` |
|-------------------------|-----------------------------|
| `full-access`           | (no `canUseTool` plumbing)  |
| `supervised` / approval-required | `default` (fires `canUseTool`) |
| plan interaction mode   | `plan`                      |
| `bypassPermissions`     | `bypassPermissions`         |

`interactionMode: plan` additionally sets the SDK mode with `setPermissionMode("plan")` at turn start (~line 3029) and reverts on completion.

### Sessions (lifecycle)

From the user clicking a row in the sidebar to the SDK running:

1. **Create thread**: UI dispatches a `thread.create` orchestration command. Decider + projector persist it; UI hydrates from the new snapshot.
2. **First user turn**: UI dispatches `thread.turn.start` with the prompt.
3. **Reactor**: `ProviderCommandReactor` consumes the resulting `thread.turn-start-requested` event and calls `ProviderService.sendTurn(...)`. If no session exists for this thread yet, `ProviderService` first calls `ClaudeAdapter.startSession`, which spawns the Claude query fiber.
4. **Streaming**: SDK messages → adapter runtime events → `ProviderRuntimeIngestion` → orchestration events → WebSocket push → UI.
5. **Interrupt**: UI dispatches `thread.turn.interrupt` → reactor calls `ClaudeAdapter.interruptTurn`.
6. **Resume after restart**: server re-creates sessions on demand with stored `resumeCursor` — the UI doesn't need to do anything special.
7. **Stop session**: happens on thread deletion (`ThreadDeletionReactor`) or manual stop — closes the prompt queue, cancels pending approvals, interrupts the fiber, removes the context.

## 4. GUI re-interpretations of "terminal" features

The recurring pattern: **a feature that would be interactive text in the terminal is modeled as an orchestration activity + a dedicated UI component.**

| Terminal feature            | Orchestration representation                    | UI location                                                                              |
|-----------------------------|-------------------------------------------------|------------------------------------------------------------------------------------------|
| Tool permission prompt      | `activity.kind === "approval.requested"`        | `ComposerPendingApprovalPanel.tsx` + `ComposerPendingApprovalActions.tsx`                |
| Tool execution + output     | `tool.started` / `tool.updated` / `tool.completed` | Timeline work-log rows in `MessagesTimeline.tsx`                                      |
| Streaming assistant text    | `thread.message.delta` (streaming mode)         | `ChatMessage.streaming: boolean`; partial text shown in-place until `streaming=false`    |
| "Press a key to continue"   | `user-input.requested` activity                 | Composer inline input prompt                                                             |
| Plan mode                   | `turn.plan.proposed`                            | Plan acceptance banner / action row                                                      |
| Session status              | `session.orchestrationStatus` + active turn id  | `deriveActiveWorkStartedAt` + working-row spinner in `MessagesTimeline.logic.ts` (~188) |
| `bash` command in workspace | Either a Claude Bash tool activity, or a direct terminal via `terminal.open` (xterm drawer) | `ThreadTerminalDrawer.tsx` (xterm + addon-fit)                     |

Two interactive UI modes for shell use deliberately coexist:
- **Agent-driven bash** — the model calls a `Bash` tool, shown as a work-log entry; the agent sees the result.
- **User-driven terminal** — `TerminalManager` spawns a PTY via `node-pty`, proxies keystrokes and output over the WebSocket `terminal.*` methods and `subscribeTerminalEvents`, renders in xterm.js. Up to four terminals per thread.

Status/quiescence: purely derived from orchestration state, not from spinners pulled off a TTY. The `working` row is appended when `isWorking === true` (derived from the session's `orchestrationStatus` and the active turn id). Reactors signal true quiescence via `RuntimeReceiptBus` (`turn.processing.quiesced`) — used by tests, not the UI.

### Checkpoints and revert

`CheckpointReactor` captures a hidden git ref before and after each turn and computes the turn diff via `CheckpointDiffQuery`. The UI shows changed-files counts on timeline rows (`assistantTurnDiffSummary`) and opens a diff panel when clicked. "Revert this turn" dispatches `thread.checkpoint.revert`, the reactor restores the git ref, and the projector truncates thread state back to that point.

## 5. Where the boundaries actually sit

Concrete entry/exit points when adding a feature:

- **Want a new user-triggered action?** Add an orchestration command in `packages/contracts/src/orchestration.ts`, teach `decider.ts` + `projector.ts`, add UI dispatch. The command goes over the one `orchestration.dispatchCommand` RPC.
- **Want to expose a new Claude Code event?** Decide whether it's a provider-runtime concept or an orchestration concept. Adapter emits a typed runtime event; `ProviderRuntimeIngestion` translates it to an orchestration command (or directly to a domain event). The UI then only has to listen for the new domain-event type.
- **Want to render a new tool type differently?** Update `classifyToolItemType` (ClaudeAdapter.ts ~395) if you need a new class; update `deriveWorkLogEntries` (`session-logic.ts` ~472) for presentation; don't touch transport.
- **Want a new subscription feed?** Define a new `Rpc.make(WS_METHODS.subscribeX, { payload, success: Stream schema })`, implement in `ws.ts`, consume via `wsRpcClient.ts` + `WsTransport.subscribe`. No custom framing.
- **Want to run non-SDK code in the agent's context?** Don't — put it in a reactor. The adapter stays thin and SDK-shaped; all side effects happen in reactors that react to domain events.

### What *not* to assume from stale docs

- `providers.startSession` / `providers.sendTurn` / `providers.respondToRequest` / `providers.stopSession` — **do not exist as RPC methods**. They exist as adapter functions; the wire uses `orchestration.dispatchCommand` with specific command payloads.
- `apps/server/src/codexAppServerManager.ts`, `apps/server/src/providerManager.ts`, `apps/server/src/wsServer.ts` — referenced in `AGENTS.md` / `.docs/architecture.md` but no longer exist. The equivalents are `provider/Layers/CodexSessionRuntime.ts` + `CodexAdapter.ts`, `provider/Layers/ProviderService.ts` + `ProviderAdapterRegistry.ts`, and `ws.ts`.

## 6. Known gaps / areas with room

From reading the adapter and reactor code:

- **No auto-recovery from process death mid-turn**: if the server crashes inside a Claude turn, the session is restarted fresh from `resumeCursor`, but the interrupted turn is not replayed.
- **Permission audit log**: approval decisions are not persisted beyond the in-memory deferred for the request.
- **MCP configuration UI**: MCP tools are rendered but there's no UI for managing MCP servers.
- **Claude Code slash commands**: listed in the composer command menu, but semantics live in the SDK/CLI. Not all SDK slash commands will present cleanly in a GUI (e.g. `/clear` semantics).
- **Tool results**: Claude tool results arrive as `user` message blocks after execution, not as streaming deltas (ClaudeAdapter.ts ~1814). Long-running tool output isn't shown in real-time — consider whether that matters for specific tools.
- **Provider coverage in docs**: `AGENTS.md` says "Codex-first" but `ClaudeAdapter`, `CursorAdapter`, and `OpenCodeAdapter` are all shipped. Worth updating.
